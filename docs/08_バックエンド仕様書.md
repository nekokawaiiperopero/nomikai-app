# バックエンド仕様書

## 1. システムアーキテクチャ

```
┌─────────────────────────────────────────────────┐
│                 ブラウザ (クライアント)              │
│  ┌──────────────────────────────────────────┐    │
│  │        app/index.html (単一HTML)          │    │
│  │  ┌────────────┐  ┌───────────────────┐   │    │
│  │  │ JavaScript │  │  Supabase JS SDK  │   │    │
│  │  │ (インライン)│  │  (CDN v2 UMD)     │   │    │
│  │  └─────┬──────┘  └────────┬──────────┘   │    │
│  └────────┼──────────────────┼──────────────┘    │
│           │                  │                    │
└───────────┼──────────────────┼────────────────────┘
            │                  │
   ┌────────▼────────┐  ┌─────▼────────────────┐
   │  OpenAI API     │  │  Supabase            │
   │  (外部サービス)  │  │  (BaaS)              │
   │                 │  │  ┌─────────────────┐  │
   │  Chat           │  │  │  REST API       │  │
   │  Completions    │  │  │  (PostgREST)    │  │
   │  gpt-4o-mini    │  │  │                 │  │
   │                 │  │  └────────┬────────┘  │
   └─────────────────┘  │  ┌───────▼────────┐  │
                        │  │  PostgreSQL     │  │
                        │  │  (データベース)  │  │
                        │  └────────────────┘  │
                        └──────────────────────┘
```

**設計方針:** サーバーレスアーキテクチャ。独自のバックエンドサーバーは存在せず、フロントエンドから直接 Supabase REST API と OpenAI API を呼び出す。

## 2. OpenAI API 呼び出しフロー

### エンドポイント

```
POST https://api.openai.com/v1/chat/completions
```

### リクエスト構成

```json
{
  "model": "gpt-4o-mini",
  "max_tokens": 1024,
  "messages": [
    { "role": "system", "content": "{動的生成されたシステムプロンプト}" },
    { "role": "user", "content": "..." },
    { "role": "assistant", "content": "..." },
    ...
  ]
}
```

### ヘッダー

```
Content-Type: application/json
Authorization: Bearer {APIキー}
```

### 呼び出し箇所と用途

| 呼び出し元関数 | プロンプト生成関数 | 用途 |
|---------------|------------------|------|
| `initHostChat()` | `hostPrompt()` | ホストチャット初回（「飲み会を企画したい」を自動送信） |
| `sendHostMsg()` | `hostPrompt()` | ホストからのメッセージに対するAI返答 |
| `startMemberChat()` | `memberPrompt(member)` | 参加者チャット初回（「こんにちは」を自動送信） |
| `sendMemberMsg()` | `memberPrompt(member)` | 参加者からのメッセージに対するAI返答 |

### API 呼び出しシーケンス（ホスト側）

```
[ホスト] → メッセージ入力
  │
  ├─ (1) addMsg: UIにユーザーメッセージ表示
  ├─ (2) hostHistory.push: メモリ上の履歴に追加
  ├─ (3) dbAddChat: Supabaseにユーザーメッセージ保存
  ├─ (4) addMsg(typing): タイピングインジケータ表示
  │
  ├─ (5) hostPrompt(): システムプロンプト生成
  │     └─ dbGetProject → dbGetMembers → dbGetNotifs
  │
  ├─ (6) callAI: OpenAI API 呼び出し
  │     └─ fetch('https://api.openai.com/v1/chat/completions')
  │
  ├─ (7) hostHistory.push: AI返答を履歴に追加
  ├─ (8) dbAddChat: SupabaseにAI返答保存
  ├─ (9) rmTyping → addMsg: タイピング除去 → AI返答表示
  ├─ (10) processHostReply: ✅マーカー検知 → DB操作
  └─ (11) renderSidebar: サイドバー更新
```

### API 呼び出しシーケンス（参加者側）

```
[参加者] → メッセージ入力
  │
  ├─ (1)-(4): ホスト側と同様
  │
  ├─ (5) memberPrompt(member): システムプロンプト生成
  │     └─ dbGetProject
  │
  ├─ (6) callAI: OpenAI API 呼び出し
  │
  ├─ (7)-(9): ホスト側と同様
  └─ (10) processMemberReply: ✅マーカー検知 → ステータス更新・通知
```

### エラーハンドリング

- APIキー未設定時: `throw new Error('APIキーが未設定')`
- APIエラー時: レスポンスボディの `error.message` を表示、取得不可時は `'API Error ' + status`
- エラーメッセージは `.msg.system` クラスでチャット内に表示

## 3. Supabase REST API 呼び出しパターン

### 接続設定

```javascript
const SUPA_URL = 'https://ypdixvgdufiwcktyzfeb.supabase.co';
const SUPA_KEY = 'sb_publishable_...';  // Publishable Key (anon)
const sb = supabase.createClient(SUPA_URL, SUPA_KEY);
```

### CRUD 操作一覧

#### projects テーブル

| 操作 | 関数 | Supabase メソッドチェーン |
|------|------|--------------------------|
| 単一取得 | `dbGetProject(id)` | `sb.from('projects').select('*').eq('id', id).single()` |
| 全件取得 | `dbGetAllProjects()` | `sb.from('projects').select('*').order('created_at', { ascending: false })` |
| 登録/更新 | `dbSaveProject(pj)` | `sb.from('projects').upsert(pj)` |
| 削除 | `dbDeleteProject(id)` | `sb.from('projects').delete().eq('id', id)` |

#### members テーブル

| 操作 | 関数 | Supabase メソッドチェーン |
|------|------|--------------------------|
| PJ別取得 | `dbGetMembers(projectId)` | `sb.from('members').select('*').eq('project_id', projectId).order('slot')` |
| 登録/更新 | `dbSaveMember(m)` | `sb.from('members').upsert(data)` |
| 削除 | `removeLastSlot()` 内 | `sb.from('members').delete().eq('id', last.id)` |
| PJ一括削除 | `dbDeleteProject(id)` 内 | `sb.from('members').delete().eq('project_id', id)` |

**dbSaveMember の特殊処理:** `phone` フィールドは `m.phone !== undefined` の場合のみ upsert データに含める。これにより、phone を意図せず null で上書きすることを防ぐ。

#### chat_messages テーブル

| 操作 | 関数 | Supabase メソッドチェーン |
|------|------|--------------------------|
| ホストチャット取得 | `dbGetChat(pjId, null)` | `.eq('project_id', pjId).is('member_id', null)` |
| 参加者チャット取得 | `dbGetChat(pjId, mid)` | `.eq('project_id', pjId).eq('member_id', mid)` |
| タイムスタンプ付き取得 | `dbGetChatWithTimestamp(pjId, mid)` | 同上（`created_at`, `member_id` を含む） |
| マージ取得 | `dbGetMergedMemberChat(pjId, mid)` | 個別 + group を2回クエリ → JS上でソート |
| 追加 | `dbAddChat(pjId, mid, role, content)` | `sb.from('chat_messages').insert(...)` |
| PJ一括削除 | `dbDeleteProject(id)` 内 | `.delete().eq('project_id', id)` |
| グループ通知追加 | `sendBroadcast()` 内 | `.insert({ member_id: 'group', ... })` |

#### notifications テーブル

| 操作 | 関数 | Supabase メソッドチェーン |
|------|------|--------------------------|
| PJ別取得 | `dbGetNotifs(projectId)` | `sb.from('notifications').select('*').eq('project_id', projectId).order('created_at')` |
| 追加 | `dbAddNotif(pjId, type, message)` | `sb.from('notifications').insert(...)` |
| 既読更新 | `dbMarkNotifsRead(pjId)` | `.update({ read: true }).eq('project_id', pjId).eq('read', false)` |
| PJ一括削除 | `dbDeleteProject(id)` 内 | `.delete().eq('project_id', id)` |

### PJ 削除時の実行順序

カスケード削除はDB側で設定済み（`on delete cascade`）だが、フロントエンドでも明示的に順次削除を行う:

```
(1) chat_messages DELETE WHERE project_id = id
(2) notifications  DELETE WHERE project_id = id
(3) members        DELETE WHERE project_id = id
(4) projects       DELETE WHERE id = id
```

## 4. データフロー

### PJ作成 → ヒアリング → 参加者登録 → 運用 → 精算

```
[PJ作成]
  ├─ projects INSERT (status='ヒアリング中', details={})
  └─ initHostChat → OpenAI API (フォーマットテンプレート提示)

[ヒアリング]
  ├─ ホスト: フォーマット入力
  ├─ AI: 「✅ 要件確定」→ details パース・保存
  │   └─ projects UPDATE (status='参加者登録中', details={name,date,...})
  └─ members INSERT × N (status='空席', スロット作成)

[参加者登録]
  ├─ ホスト: 参加者名を入力
  ├─ AI: 「✅ 参加者登録: A, B, C」
  │   └─ projects UPDATE (status='運用中')
  └─ members UPDATE (name, status='未回答', invited_at, expires_at)

[運用]
  ├─ 参加者がURL経由でアクセス
  │   ├─ 期限チェック (expires_at vs now)
  │   │   ├─ 期限内 → 参加者チャット開始
  │   │   └─ 期限切れ → S5画面 + members UPDATE (status='期限切れ')
  │   ├─ AI: 飲み会詳細を表示、参加可否を確認
  │   ├─ 参加者: 回答
  │   │   ├─ 「✅ 参加OK」→ members UPDATE (status='参加')
  │   │   ├─ 「✅ 不参加」→ members UPDATE (status='不参加')
  │   │   └─ 「✅ キャンセル」→ members UPDATE (status='キャンセル', cancel_fee=予算)
  │   ├─ notifications INSERT (各イベント)
  │   └─ chat_messages INSERT (ホストチャットに📢通知)
  │
  ├─ ホスト: ダッシュボード操作
  │   ├─ 一斉通知 → chat_messages INSERT (member_id='group')
  │   ├─ 催促 → chat_messages INSERT (参加者個別チャット)
  │   ├─ 再発行 → members UPDATE (expires_at を24h延長)
  │   ├─ 枠追加/削除 → members INSERT/DELETE
  │   └─ 名前変更 → members UPDATE
  │
  └─ ホスト: 変更通知/中止
      ├─ 「✅ 変更通知」→ 全参加者にブロードキャスト
      └─ 「✅ 中止」→ projects UPDATE (status='中止') + 全参加者に中止通知

[精算]
  ├─ 実費入力: localStorage に保存 ('settle-{pjId}')
  └─ 割り勘計算: Math.ceil(実費 / 参加確定人数)
```

### 一斉通知（ブロードキャスト）フロー

```
ホスト → broadcastInput に入力 → 「送信」ボタン
  │
  ├─ chat_messages INSERT (member_id='group', role='assistant', content=テキスト)
  ├─ chat_messages INSERT (member_id=null, role='assistant', content='📢 全員への通知: ...')
  ├─ notifications INSERT (type='broadcast')
  └─ ホストチャットUI更新 (notification メッセージ追加)

参加者がチャット画面を開いた時:
  └─ dbGetMergedMemberChat: 個別メッセージ + group メッセージを時系列マージ
      └─ group メッセージは「📢 幹事から: {内容}」として group-msg クラスで表示
```

### 通知連動フロー

```
参加者チャットでイベント検知（✅マーカー）
  │
  ├─ (1) members UPDATE (ステータス変更)
  ├─ (2) notifications INSERT (type=join/decline/cancel/allergy/late/plus_one/phone)
  ├─ (3) chat_messages INSERT (member_id=null, '📢 {hostMsg}')
  └─ (4) updateBadge() (未読通知数をバッジに反映)
```

## 5. 認証・セキュリティ

### OpenAI APIキー管理

| 項目 | 内容 |
|------|------|
| 保存先 | `localStorage` (キー: `nomikai-openai-key`) |
| 入力UI | トップ画面の `input[type=password]` |
| 読み込み優先順 | 入力欄の値 → localStorage の値 |
| 露出 | フロントエンドからの直接API呼び出しのため、ブラウザDevToolsで閲覧可能 |
| 対策 | ハッカソンデモ用のため割り切り。本番ではサーバーサイド経由が必須 |

### Supabase 認証

| 項目 | 内容 |
|------|------|
| 認証方式 | なし（Supabase Auth 未使用） |
| API Key | Publishable Key (anon) をフロントエンドにハードコード |
| RLS | 全テーブル `allow all` ポリシー（デモ用） |

### RLS ポリシー

全4テーブルに同一の `allow all` ポリシーが設定されている:

```sql
CREATE POLICY "allow all {table}" ON {table}
  FOR ALL USING (true) WITH CHECK (true);
```

**本番環境への移行時に必要な対策:**
- Supabase Auth の導入（ユーザー認証）
- RLS ポリシーの厳格化（`auth.uid()` に基づくアクセス制御）
- APIキーのサーバーサイド管理（Edge Functions等）

## 6. エラーハンドリング方針

### OpenAI API エラー

| エラー種別 | ハンドリング |
|-----------|-------------|
| APIキー未設定 | `alert('OpenAI API Keyを入力してください')` (PJ作成時) / `throw new Error('APIキーが未設定')` (チャット時) |
| API レスポンスエラー (4xx/5xx) | レスポンスJSONの `error.message` を取得。取得不可時は `'API Error ' + status` |
| ネットワークエラー | `catch` ブロックで `e.message` を表示 |
| 表示方法 | チャット画面内に `.msg.system` クラスで `'エラー: ' + e.message` を表示 |

### Supabase API エラー

| エラー種別 | ハンドリング |
|-----------|-------------|
| クエリエラー | `{ data }` を分割代入で取得。data が null の場合は空配列 `[]` にフォールバック |
| 接続エラー | 暗黙的にエラーが無視される（エラーUI表示なし） |

### UI エラー

| エラー種別 | ハンドリング |
|-----------|-------------|
| 無効な参加者URL | `alert('無効なURLです')` |
| クリップボードAPI失敗 | `prompt('コピー:', text)` でフォールバック |
| 二重送信 | `sending` フラグで防止、送信ボタンを `disabled` に設定 |

### 全体方針

- **ユーザーへのフィードバック:** エラーはチャット内のシステムメッセージまたは `alert` で表示
- **グレースフルデグラデーション:** Supabase クエリ結果が null の場合は空配列にフォールバック
- **リトライなし:** API 呼び出しのリトライ機構は未実装
- **ログ出力なし:** `console.log` / `console.error` によるデバッグ出力は未実装
