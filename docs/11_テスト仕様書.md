# テスト仕様書

本ドキュメントは「飲み会エージェント」の各機能に対する動作確認手順を定義する。

---

## テスト環境

| 項目 | 内容 |
|------|------|
| ブラウザ | Chrome最新版（推奨）、Safari、Firefox |
| デバイス | PC（Chrome）+ スマートフォン（参加者画面の確認用） |
| OpenAI APIキー | 有効なキーを事前に取得しておくこと |
| Supabase | テーブル作成・RLSポリシー設定済みであること |

---

## TC-1: PJ作成とAIヒアリング

### TC-1-1: 新規PJ作成

| 項目 | 内容 |
|------|------|
| 前提条件 | トップ画面（S1）が表示されている。OpenAI APIキーが入力済み |
| 操作手順 | 1. 「新しい飲み会を作る」ボタンをクリック |
| 期待結果 | ホストチャット画面（S2）に遷移し、AIがフォーマットテンプレート（飲み会名・日時・場所・予算・人数・備考）を提示する |

### TC-1-2: フォーマット入力による要件確定

| 項目 | 内容 |
|------|------|
| 前提条件 | TC-1-1 完了後、AIがフォーマットテンプレートを提示している |
| 操作手順 | 1. フォーマットの各項目を埋めて送信する（例: 飲み会名「歓迎会」、日時「2026-04-01 19:00」、場所「渋谷 居酒屋ABC」、予算「5000円」、人数「4人」） |
| 期待結果 | AIが入力内容を確認し「要件確定」メッセージを出力する。コピーボタン付きの確定フォーマットが表示される。Supabase の `projects` テーブルに新レコードが作成される |

### TC-1-3: 参加者名ヒアリング

| 項目 | 内容 |
|------|------|
| 前提条件 | TC-1-2 完了後、要件が確定している |
| 操作手順 | 1. AIの質問に応じて参加者名を入力する（例:「田中、佐藤、鈴木、高橋」） |
| 期待結果 | AIが「参加者登録: 田中, 佐藤, 鈴木, 高橋」と確認メッセージを出力する。`members` テーブルに4件のレコードが作成される |

### TC-1-4: APIキー未入力時のエラー

| 項目 | 内容 |
|------|------|
| 前提条件 | OpenAI APIキーが未入力の状態 |
| 操作手順 | 1. 「新しい飲み会を作る」ボタンをクリック |
| 期待結果 | APIキーの入力を促すメッセージが表示される、またはチャットでAI応答がエラーとなる |

## TC-2: 参加者登録

### TC-2-1: チャット経由での参加者登録

| 項目 | 内容 |
|------|------|
| 前提条件 | PJが作成済みで、ホストチャット画面（S2）が表示されている |
| 操作手順 | 1. AIヒアリング中に参加者名を伝える（例:「参加者は山田、田中、佐藤の3人です」） |
| 期待結果 | AIが参加者名を確認し、`members` テーブルに参加者レコードが作成される |

### TC-2-2: ダッシュボード経由での参加者登録

| 項目 | 内容 |
|------|------|
| 前提条件 | PJが作成済みで、ダッシュボード画面（S3b）の参加者タブが表示されている。空きスロットが存在する |
| 操作手順 | 1. 空きスロットの名前入力欄に参加者名を入力する（例:「山田太郎」） 2. 入力欄からフォーカスを外す |
| 期待結果 | スロットに名前が登録される。`members` テーブルの該当レコードの `name` が更新される |

### TC-2-3: 枠の追加

| 項目 | 内容 |
|------|------|
| 前提条件 | ダッシュボード画面（S3b）の参加者タブが表示されている |
| 操作手順 | 1. 「+ 枠を追加」ボタンをクリック |
| 期待結果 | スロット一覧の末尾に新しい空きスロットが追加される。`members` テーブルに新レコードが追加される |

### TC-2-4: 枠の削除

| 項目 | 内容 |
|------|------|
| 前提条件 | ダッシュボード画面（S3b）の参加者タブにスロットが複数存在する |
| 操作手順 | 1. 「- 最後の枠を削除」ボタンをクリック 2. 確認ダイアログで「OK」を押す |
| 期待結果 | 最後のスロットが削除される。`members` テーブルから該当レコードが削除される |

## TC-3: 招待リンク

### TC-3-1: 招待リンクの発行とコピー

| 項目 | 内容 |
|------|------|
| 前提条件 | ダッシュボード画面（S3b）で参加者が登録されている |
| 操作手順 | 1. 対象参加者の「URLコピー」ボタンをクリック |
| 期待結果 | クリップボードに招待リンクがコピーされる。リンク形式は `{公開URL}#join/{PJ_ID}/{参加者ID}`。ボタン表示が一時的に「コピー済」に変わる |

### TC-3-2: 招待リンクからのアクセス（有効期限内）

| 項目 | 内容 |
|------|------|
| 前提条件 | TC-3-1 でコピーした招待リンクが有効期限内（発行から24時間以内） |
| 操作手順 | 1. 招待リンクをブラウザで開く |
| 期待結果 | 参加者チャット画面（S4）が表示される。AIが飲み会の詳細情報をフォーマットで提示し、参加可否を確認する |

### TC-3-3: 招待リンクの期限切れ

| 項目 | 内容 |
|------|------|
| 前提条件 | 招待リンクの有効期限（`expires_at`）が過去の日時に設定されている（DB直接編集でテスト可能） |
| 操作手順 | 1. 期限切れの招待リンクをブラウザで開く |
| 期待結果 | 期限切れ画面（S5）が表示される。参加者チャット画面には遷移しない |

### TC-3-4: 招待リンクの再発行

| 項目 | 内容 |
|------|------|
| 前提条件 | ダッシュボード画面（S3b）で、ある参加者の招待リンクが期限切れ状態 |
| 操作手順 | 1. 対象参加者の「再発行」ボタンをクリック |
| 期待結果 | `members` テーブルの `invited_at` と `expires_at` が更新される。新しいリンクで再度アクセスできるようになる。通知テーブルに再発行の記録が追加される |

## TC-5: ダッシュボード

### TC-5-1: ステータス反映の確認

| 項目 | 内容 |
|------|------|
| 前提条件 | 参加者が招待リンク経由で参加/不参加/キャンセルを回答済み |
| 操作手順 | 1. ダッシュボード画面（S3b）の参加者タブを開く |
| 期待結果 | 各参加者のステータスバッジが最新の状態を反映している（参加=緑、不参加=赤、キャンセル=赤、未回答=黄、空席=グレー） |

### TC-5-2: 通知バッジの表示

| 項目 | 内容 |
|------|------|
| 前提条件 | 参加者が何らかのアクション（参加表明等）を行い、未読通知が存在する |
| 操作手順 | 1. ホストチャット画面（S2）のヘッダーにある「管理」ボタンを確認する |
| 期待結果 | 「管理」ボタンに未読通知数のバッジ（赤い丸数字）が表示される |

### TC-5-3: 通知タブの確認

| 項目 | 内容 |
|------|------|
| 前提条件 | 何らかの通知イベントが発生済み |
| 操作手順 | 1. ダッシュボード画面（S3b）の「通知」タブをクリック |
| 期待結果 | 通知が時系列順で表示される。未読通知は強調表示される。通知タブを開くと既読に切り替わる |

### TC-5-4: 精算タブの確認

| 項目 | 内容 |
|------|------|
| 前提条件 | PJに参加者が登録されており、参加/キャンセルのステータスが混在する |
| 操作手順 | 1. ダッシュボード画面（S3b）の「精算」タブをクリック 2. 実費の金額を入力する |
| 期待結果 | 見込み合計、割り勘金額、キャンセル料一覧が正しく計算・表示される |

### TC-5-5: 催促ボタン

| 項目 | 内容 |
|------|------|
| 前提条件 | ダッシュボード画面（S3b）で未回答の参加者が存在する |
| 操作手順 | 1. 対象参加者の「催促」ボタンをクリック |
| 期待結果 | 参加者のチャットにリマインドメッセージが投稿される。`notifications` テーブルに催促送信の通知が追加される |
