# 要件定義書

## 1. システム概要

飲み会の幹事（ホスト）がAIエージェントと対話して飲み会を企画し、
参加者への連絡・やりとりをAIに委任するWebアプリケーション。

## 2. アクター定義

| アクター | 説明 |
|----------|------|
| ホスト | 飲み会を企画・管理する人。AIと対話しPJを作成する |
| 参加者 | URLを受け取り、AIとやりとりして参加可否を回答する人 |
| AIエージェント | ホストの代理として参加者とやりとりするAI |

## 3. 画面一覧

| # | 画面名 | アクター | 概要 |
|---|--------|----------|------|
| S1 | トップ画面 | ホスト | 新規PJ作成・過去PJ一覧の入口 |
| S2 | ホストチャット画面 | ホスト | AIと対話して飲み会の要件を確定する |
| S3 | ダッシュボード画面 | ホスト | 参加者一覧・状況・通知・URL管理 |
| S4 | 参加者チャット画面 | 参加者 | AIと対話して参加可否回答・質問・キャンセル |

## 4. 機能要件

### F1: 飲み会PJ作成

| 項目 | 内容 |
|------|------|
| 概要 | ホストが新しい飲み会PJを作成する |
| トリガー | トップ画面で「新しい飲み会を作る」ボタン押下 |
| 処理 | PJを新規作成し、ホストチャット画面に遷移 |
| データ | PJ ID（自動生成）、作成日時 |

### F2: ホスト-AIチャット（要件確定）

| 項目 | 内容 |
|------|------|
| 概要 | AIがホストにヒアリングし、飲み会の詳細を確定する |
| トリガー | PJ作成後、ホストチャット画面が開かれる |
| AI動作 | 以下の項目を順にヒアリングする |

**ヒアリング項目:**

| # | 項目 | 必須/任意 | 例 |
|---|------|-----------|-----|
| 1 | 飲み会名 | 必須 | 「金曜のサシ飲み」 |
| 2 | 日時 | 必須 | 「3/28(金) 19:00〜」 |
| 3 | 場所・店名 | 必須 | 「渋谷 鳥貴族」 |
| 4 | 予算（想定金額） | 必須 | 「4,000円」（キャンセル料算出に使用） |
| 5 | 予定人数 | 任意 | 「5人くらい」 |
| 6 | 参加者上限 | 任意 | 「最大8人」 |
| 7 | 備考 | 任意 | 「アレルギーある人は教えて」 |

**AI動作ルール:**
- 自然な会話形式でヒアリングする（フォーム的に聞かない）
- 必須項目が埋まったら確認メッセージを表示する
- ホストが「OK」等で承認したら要件確定とする
- 確定後、参加者登録フェーズに移行する

### F3: 参加者登録（インビテーション管理）

| 項目 | 内容 |
|------|------|
| 概要 | ホストが参加者を登録し、招待リンクを発行する |
| トリガー | 飲み会の要件が確定した後 |
| 登録方法 | AIチャットまたはダッシュボードから登録 |

**招待リンクの仕様:**
- URL形式: `{アプリURL}#join/{PJ_ID}/{参加者ID}`
- 参加者IDはランダム文字列
- **有効期限: 発行から24時間**
- 期限切れURLにアクセスした場合、期限切れメッセージを表示

**仮名登録:**
- 参加者名が未確定でも仮名（例：「参加者1」「友人A」）でリンク発行可能
- 後からチャットまたはダッシュボードで氏名を変更可能

### F4: ダッシュボード

| 項目 | 内容 |
|------|------|
| 概要 | ホストが参加者全員の状況を一覧で確認・管理する |

**ダッシュボード表示項目:**

| 項目 | 説明 |
|------|------|
| 参加者名 | 登録された名前（編集可能） |
| ステータス | 参加 / 不参加 / キャンセル / 未回答 |
| 招待リンク | 参加者に送るURL（コピーボタン付き） |
| 招待有効期限 | 残り時間 or 期限切れ表示 |
| 最終更新 | 参加者が最後にやりとりした日時 |

**通知機能:**
- ダッシュボードに未読通知バッジを表示
- ホストチャット画面のダッシュボードボタンにも通知バッジ表示
- 通知対象イベント:
  - 参加者が参加を表明
  - 参加者が不参加を表明
  - 参加者がキャンセルした
  - 招待リンクの期限切れ
  - アレルギー・食事制限の申告
  - 遅刻連絡
  - 追加参加者の申請

**通知ログ:**
- 通知一覧を時系列で表示するセクション
- 例：「[14:30] 田中さんが参加を表明しました」「[15:00] 佐藤さんがキャンセルしました（キャンセル料: ¥4,000）」

**ホスト操作:**
- 参加者名の編集（仮名→本名への変更）
- 参加者の追加登録
- 未回答者への催促（AIがリマインドメッセージを投稿）
- 飲み会詳細の変更 → 参加者全員に変更通知
- 飲み会の中止通知
- 参加確定者への一斉メッセージ送信

**サマリ表示:**
- 参加 / 不参加 / キャンセル / 未回答の人数
- 飲み会概要（日時・場所・予算等）
- 定員に対する残り枠数（上限設定時）

### F5: 参加者-AIチャット

| 項目 | 内容 |
|------|------|
| 概要 | 参加者がURLを開き、AIとやりとりする |
| トリガー | 参加者が招待URLにアクセス |

**招待URL検証:**
- URLの有効性チェック（存在するPJ・参加者か）
- 有効期限チェック（24時間以内か）
- 期限切れの場合、「この招待リンクは期限切れです。幹事にお問い合わせください。」を表示

**AI動作フロー:**
1. 「{参加者名}さん、こんにちは！{飲み会名}のお誘いです！」と挨拶
2. 飲み会の詳細（日時・場所・予算等）を表示
3. 参加可否を確認
4. 参加者からの質問に回答（飲み会情報の範囲内）
5. 回答結果をダッシュボード・通知に反映

**参加者ができること:**
- 参加 / 不参加を回答
- 参加後にキャンセル
- 飲み会について質問する（場所の詳細、予算、ドレスコード等）
- アレルギー・食事制限の申告
- 遅刻連絡（到着予定時刻の変更）
- 追加参加者の申請（「友達も連れて行っていい？」）
- 回答を変更する

### F6: キャンセル処理

| 項目 | 内容 |
|------|------|
| 概要 | 参加確定後のキャンセルに対し、キャンセル料を通知する |
| トリガー | 参加者がキャンセルの意思をAIに伝える |

**キャンセル料ルール:**
- キャンセル料 = 想定金額（予算）の100%
- AIが「キャンセル料として¥{金額}が発生します。よろしいですか？」と確認
- 参加者が承諾したらキャンセル確定
- ホストに通知:「{参加者名}さんがキャンセルしました（キャンセル料: ¥{金額}）」

### F7: リマインド

| 項目 | 内容 |
|------|------|
| 概要 | 飲み会前日に参加者へリマインドを送る |

**リマインド仕様:**
- 飲み会の前日時点で、参加者チャットにAIがリマインドメッセージを自動投稿
- 内容：「明日は{飲み会名}です！{場所}に{時間}集合です 🎉」
- 未回答者にはリマインド＋参加可否の再確認
- 実装：ホストがダッシュボードを開いた時 or チャットを開いた時に前日判定でリマインド投稿

**手動リマインド:**
- ホストがダッシュボードから未回答者への催促ボタンを押す
- AIが参加者チャットにリマインドメッセージを投稿

### F8: 飲み会詳細の変更通知

| 項目 | 内容 |
|------|------|
| 概要 | 飲み会の詳細が変わった場合、全参加者に通知する |
| トリガー | ホストがチャットで「場所変わった」等の変更を伝える |

**変更通知フロー:**
1. ホストがAIに変更内容を伝える
2. AIが変更内容を確認・確定
3. 全参加者のチャット履歴に変更通知メッセージを追加
4. 参加者が次回チャットを開いた時に変更通知が表示される

### F9: 飲み会中止

| 項目 | 内容 |
|------|------|
| 概要 | ホストが飲み会を中止する |
| トリガー | ホストがチャットで中止を伝える or ダッシュボードから中止操作 |

**中止フロー:**
1. AIが確認:「飲み会を中止しますか？」
2. ホスト承認後、全参加者のチャットに中止通知を追加
3. PJステータスを「中止」に変更
4. 以降の参加者チャットでは中止メッセージを表示

### F10: 精算サマリ

| 項目 | 内容 |
|------|------|
| 概要 | 飲み会終了後に精算の概要を表示する |
| 表示場所 | ダッシュボード |

**精算表示項目:**
- 参加者数 × 想定金額 = 合計見込み金額
- キャンセル者とキャンセル料一覧
- 実費入力欄（ホストが実際の金額を入力）
- 割り勘金額の自動計算（実費 ÷ 参加者数）

## 5. データ設計

### localStorageキー構造

```
nomikai-app-projects          → PJ一覧 [{id, name, createdAt}]
nomikai-app-pj-{PJ_ID}       → PJ詳細 {name, date, place, budget, headcount, maxMembers, note, status}
nomikai-app-members-{PJ_ID}  → 参加者一覧 [{id, name, status, invitedAt, expiresAt, cancelFee, lastUpdated}]
nomikai-app-chat-host-{PJ_ID}      → ホストチャット履歴
nomikai-app-chat-member-{PJ_ID}-{MEMBER_ID} → 参加者チャット履歴
nomikai-app-notifications-{PJ_ID}  → 通知ログ [{timestamp, type, message, read}]
nomikai-app-settlement-{PJ_ID}     → 精算データ {actualCost, perPerson, cancelFees}
```

### 参加者ステータス遷移

```
未回答 → 参加 → キャンセル
未回答 → 不参加
```

### PJステータス遷移

```
作成 → ヒアリング中 → 要件確定 → 参加者登録中 → 運用中 → 終了
                                                     └→ 中止
```

### 通知タイプ一覧

| type | 表示例 |
|------|--------|
| join | 「{name}さんが参加を表明しました」 |
| decline | 「{name}さんが不参加を表明しました」 |
| cancel | 「{name}さんがキャンセルしました（キャンセル料: ¥{amount}）」 |
| expired | 「{name}さんの招待リンクが期限切れになりました」 |
| allergy | 「{name}さんがアレルギー・食事制限を申告しました: {detail}」 |
| late | 「{name}さんが遅刻連絡をしました: {detail}」 |
| plus_one | 「{name}さんが追加参加者を申請しました: {detail}」 |
| detail_change | 「飲み会の詳細が変更されました」 |
| cancelled_event | 「飲み会が中止されました」 |

## 6. 非機能要件

| 項目 | 要件 |
|------|------|
| レスポンシブ | モバイルファーストで設計（参加者はスマホでアクセスする想定） |
| データ保存 | localStorage（サーバー不要） |
| AI | OpenAI API (gpt-4o-mini) を使用 |
| URL共有 | ホストがURLをコピーし、既存SNS/メッセンジャーで手動送付 |
| ブラウザ | Chrome / Safari 最新版 |

```

    |-- 「新しい飲み会を作る」
    v
[S2: ホストチャット画面] <-- 通知バッジ付きダッシュボードボタン
```

## 8. 技術上の制約・注意点

1. **localStorage の制約**
   - ブラウザ・端末をまたいだデータ共有はできない
   - ホストと参加者は同じ端末で動作する前提（デモ用）
   - 本番運用にはバックエンド（DB + API）が必要

2. **OpenAI API の呼び出し**
   - フロントエンドから直接APIを呼ぶ場合、APIキーが露出する
   - ハッカソンデモでは許容するが、本番ではバックエンドが必要
   - APIキーはアプリ起動時に入力させる

3. **URL共有の制約**
   - ローカルサーバー（localhost）ではURL共有が機能しない
   - デモでは同一ブラウザの別タブで参加者画面を開く想定

4. **リマインドの制約**
   - サーバーレスのため自動リマインドは不可
   - ホストがアプリを開いた時に前日判定でリマインド投稿する疑似実装
   - 手動催促ボタンで代替

5. **通知の制約**
   - プッシュ通知なし（ブラウザ通知も未実装）
   - ダッシュボードを開いた時にのみ通知が更新される
